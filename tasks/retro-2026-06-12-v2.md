# Retro (/sk:retro) — Vibe Builder v2

## What shipped
3 features: custom color palette, production-app mode + stack recommendation,
ShipKit-powered app generation. Plus concurrency-guard (409). Zero v1 invariant
regressions; real app-mode generation verified end-to-end.

## Velocity
- Single pass, no rework loops. Backend-first (with a 19-assert unit harness
  before touching the frontend) caught all logic issues pre-integration.
- Real headless gen completed in ~one run on haiku; produced the ShipKit plan
  artifact as designed.

## What went well
- Splitting prompt.js into static vs app builders kept static mode byte-identical
  (clean regression) while isolating app complexity.
- Validating + asserting the assembled prompt string (palette verbatim, ShipKit
  discipline present) gave fast, deterministic proof without burning gen calls.
- Concurrency lock as an in-memory Set keyed by project id: 12 lines, 409 verified.

## Blockers / friction
- curl blocked by sandbox hook -> switched smoke tests to Node fetch.
- Background server via plain `&` didn't persist -> used run_in_background.
- Haiku exhausts turns on full Laravel scaffold (composer create-project slow).
  Honest limitation documented; plan artifact still reliably produced.

## Improvements for next time
- Consider a thin "scaffold first via a fixed composer/npx step, THEN hand to the
  child" path for app mode so the model spends turns on app logic, not framework
  bootstrap. Would make Haiku app-mode reliably complete.
- Optional: opt-in auto-start of the app dev server behind the existing timeout/
  SIGKILL process discipline, pointing the iframe at the captured port.
